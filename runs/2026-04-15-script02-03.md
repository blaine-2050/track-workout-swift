## Bugs fixed this session

1. **Sticky-input flag was global, not per-field** (`WorkoutViewModel.isSticky`). Typing the first digit in *one* field consumed the sticky replacement on *both* fields — so in a multi-set workflow where the user enters a new weight and then taps reps, the reps sticky was already disarmed and the new digit appended instead of replacing.
   - Fix: split into `isWeightSticky` and `isRepsSticky`; each cleared independently on its field's first keypress.
   - This wasn't caught by script 02 because script 02's reps-sticky test was the first action after Log, not after weight entry. Script 03 reproduced it with `335` then reps `2` → became `32`.

2. **Keypad accessibility.** Added `accessibilityIdentifier("keypad-N")` to all 12 keypad buttons. Without this, tests couldn't target digits unambiguously because e.g. "5" also matched the Reps display when reps=5 (sticky).

## Bugs found but NOT fixed (captured for followup)

- **"Sets" terminology disagreement.** Modal summary shows `3 sets` for 3 logged entries of Deadlift. Inline-history summary shows `Sets: 1` — the inline code groups consecutive entries of the same move as one "Set" (with 3 sub-entries). Both are internally consistent, but the PRD/script 03 expects `3 sets`. Options:
  1. Change inline summary to count entries (agrees with modal/spec).
  2. Rename inline "Set N" groupings to something else ("Move block N"?) and keep "Sets" for entry count.
  3. Update the spec to disambiguate.
  - Spec question for user. For now, the flow asserts `Total: 2,560` (which is unambiguous) instead of `Sets: N`.
- **Maestro same-element tap coalescence.** Consecutive text-based taps on the same element collapse into one action. Workaround: use accessibility IDs (now in place) or point-based taps.
- **Maestro full-string regex.** `assertVisible` uses full-string regex match — `"foo"` won't match `"foo bar"`; need `"foo.*"`. Now standard in all our flows.
